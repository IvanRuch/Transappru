- [ ] `[transapp • resilience]` Backend latency spike'и до 20s на `/get-auto-list` (Иван). Корень — НЕ client-specific: данные клиента 474 (ЭКО-СТРОЙ, 6 авто, 350 fine / 1914 avtodor) средние по сравнению с другими firm того же телефона (Вектура: 74 авто / 4389 fine / 50311 avtodor — летает; Дорожные Плиты: 14 авто / 1566 avtodor — летает). Гипотеза «много истории у клиента» проверена через DataGrip на prod-БД 2026-05-25 и отвергнута. По `user_auto.check_*_last_time` видны cron'ы воркеров обновления данных: `check_avtodor_last_time` ровно `10:00:03` у всех авто (1.5M записей в `user_auto_avtodor`), `check_rnis_last_time` ровно `15:00:02` — вероятная причина spike'ов — long-running транзакции этих воркеров на больших таблицах, держащие lock-wait для конкурирующих SELECT'ов из handler'а `/get-auto-list`. Конкретные шаги Ивану: (1) разбить batch-update'ы avtodor/rnis на чанки 1-5k с COMMIT после каждого; (2) или сместить cron'ы в ночное окно; (3) включить slow-query log в часы cron'ов чтобы подтвердить lock-wait root cause. Источник схемы — `legacy/SHOW CREATE TABLE.txt`. Клиентская сторона сделана максимум (ADR-023 + ADR-024 + ADR-026) 🔼 #project/transapp #epic/resilience #waiting/external-dependency
