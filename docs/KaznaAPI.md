# Общие сведения

Процессинговый центр предоставляет программный интерфейс к Сервису поиска и оплаты Начислений через Витрины Партнеров доступными в Процессинговом центре способами оплаты.

Взаимодействие осуществляется по протоколу REST API с обменом данными в формате

JSON.

Аутентификация Партнеров осуществляется на основе токена (на тестовой зоне - testservice), передаваемого в теге Authorization c типом Bearer, который выдается при подключении к сервису. Консистентность запроса на оплату проверяется на основе sign, вычисленному с использованием секретного слова (на тестовой зоне - SA9QXHKV).

Наименования полей регистрозависимые, при подсчете sign необходимо соблюдать порядок полей согласно описания. При наличии объектов для конкатенации используются значения всех параметров из объекта.



При возникновении ошибки при обработке запроса в ответе будет присутствовать структура

error с описанием (см. [тип error](#_bookmark128)).

## Уведомление об успешном платеже notify

Уведомление отправляется на url с авторизационным токеном, которые должны быть предоставлены Партнером для отправки Сервисом уведомлений. Информация о Платеже идентична ответу на запрос paymentInfo + sign (формируется как строка md5 от конкатенации значений следующих параметров: токен партнера, paymentID, status.name; пример - a3b1f52f7b6c1d864ad7623023500f5a).

Доставка информации о конечном статусе платежа по notify не является гарантированной, для получения статуса платежа необходимо использовать метод [paymentinfo](#_bookmark41).



Схема отправки уведомления, в случае не успешной доставки:

* сразу после наступления события;
* через 1 секунду;
* через 10 сек;
* через 1 мин;
* через 5 мин;
* через 20 мин;
* через 60 мин.

## Метод refund

Полный или частичный возврат денежных средств Плательщику по платежу, доступен только для Партнеров являющихся непосредственными получателями денежных средств. Возврат осуществляется взаимозачетом. Если в течении трех рабочих дней в пользу получателя не будет достаточного объема платежей для компенсации суммы возврата, то будет выставлено требование на погашение задолженности. Доступен для платежей, проведенных через способ оплаты sbp. Операция возврат производится асинхронно, результат проведения возврата необходимо запросить вызовом метода [paymentInfo](#_bookmark41) указав в качестве идентификатора paymentID, также доступен сервис получения уведомлений о изменении стауса платежа

Адрес запроса: POST https://demopay.oplatagosuslug.ru/api/kazna/2.0/refund

### Схема взаимодействия

**Полный или частичный возврат денежных средств:**

### Параметры

**Входные параметры:**

| **Название** | **Значение** | **Об.** | **Описание** |
| paymentID | string | да | Уникальный идентификатор платежа, по которому производится возврат, полученный в ответ на запрос [pay](#_bookmark8) по успешному платежу. |

|  |  |  |  |

| **Название** | **Значение** | **Об.** | **Описание** |

## Криптограмма

Возможность проведения платежей через криптограмму согласовывается с менеджером. Для формирования криптограммы должен использоваться скрипт с сайта Сервиса, ссылка на скрипт выдается Партнеру при подключении к Сервису. Скрипт размещается на витрине Партнера в целях формирования криптограммы алгоритмом RSA.

Криптограмма передается по защищенному каналу (SSL) на сервер Партнера, затем передается в Сервис, в запросе pay. Взаимодействие витрины Партнера с Сервисом напрямую запрещено.

**Требования к форме ввода данных:**

* + - У полей с карточными данными не должно быть атрибута "name", чтобы данные не попали на сервер при отправке формы.
    - Данные из формы должны передаваться по SSL.

**Требования к криптограмме:**

* + - Должна формироваться оригинальным скриптом, загруженным с сайта Сервиса
    - Криптограмму нельзя сохранять и использовать повторно

**Требования к безопасности по PCI DSS:**

С точки зрения PCI DSS, подобный способ подключения классифицируется как "E- commerce merchants who outsource all payment processing to PCI DSS validated third parties, and who have a website(s) that doesn’t directly receive cardholder data but that can impact the security of the payment transaction. No electronic storage, processing, or transmission of any cardholder data on the merchant’s systems or premises.", то есть обработка платежных данных выполняется третьей стороной, но сайт влияет на безопасность карточных данных.

Для соблюдения требований стандарта, необходимо заполнять лист самооценки SAQ-EP и ежеквартально проходить ASV тестирование. Сканирование должно проводится аккредитованным вендором (ASV) из списка, представленного на сайте совета PCI.

## Сценарии оплаты при отмене Пользователем рекуррентных платежей (ошибки и статусы)

Пользователь, выдавший разрешение Системе на проведение рекуррентных платежей, может его отменить в своём банковском приложении. В таком случае Система узнает об отмене разрешения только на этапе попытки проведения рекуррентного платежа.

Поэтому, в зависимости от номера попытки проведения рекуррентного платежа, существует два сценария поведения Системы.

### Первая попытка проведения рекуррентного после отмены разрешения пользователем



На данном этапе Система не знает, отменил ли пользователь разрешение на проведение

рекуррентных платежей.

**Шаг 1.** Партнёр выполняет вызов метода /[recurrent](#_bookmark66).

**Шаг 2.** Система отвечает успехом, платёж проинициализирован. В ответе отдаётся идентификатор платежа.

Пример ответа:

{

"paymentID": 511147

}

**Ответ на запрос recurrent**

**Шаг 3.** Партнёр выполняет вызов метода /[paymentInfo](#_bookmark41).

**Шаг 4.** Система в ответе отдаёт структуру с информацией о платеже, где указаны:

1. Статус платежа (в текущем сценарии - CANCEL);
2. Код причины отмены платежа (в текущем сценарии - 10111) - подробнее - в разделе [Причины отмены платежа](#_bookmark134);
3. Описание причины отмены платежа (в текущем сценарии "Подписка отменена") подробнее - в разделе [Причины отмены платежа](#_bookmark134)

Пример ответа:

{

"paymentID": 511147,

"orderID": "baa52e5a-44b6-8df2-e15cbc294532", "status": {

"code": "30",

"name": "cancel",

"date": "2025-08-11T18:26:57+03:00",

"cancelCode": 10111,

"cancelReason": "Подписка отменена."

},

"depType": "merchant9999999999", "kvit": true,

"amount": 150000,

"totalSum": 150000, "payType": "sbp", "paymentParams": {

"purpose": "месячная подписка Сервиса"

},

"result": [

{

"orderID": "baa52e5a-44b6-8df2-e15cbc294532", "amount": "1500.00",

"Purpose": "месячная подписка Сервиса", "Email": "test@email.com"

}

],

"sbpSubscription": {

"subscriptionUuid": "01f05eba-6779-a8ff-000c56d76ec7"

}

}

**Ответ на запрос paymentInfo**

### Вторая и последующая попытки проведения рекуррентного после отмены разрешения пользователем



На данном этапе Система уже знает, что пользователь отменил разрешение на проведение

рекуррентных платежей.

**Шаг 1.** Партнёр выполняет вызов метода /[recurrent](#_bookmark66).

**Шаг 2.** Система отвечает ошибкой, платёж не проинициализирован. В ответе отдаётся информация об ошибке (подробнее - в разделе " [ошибки при обработке запроса (error)](#_bookmark128)".

Пример ответа:

{

"code": 155,

"message": "Подписка отменена."

}

**Ответ на повторный запрос recurrent**

# Отчетность по успешным платежам

При необходимости получения ежедневной или ежемесячной отчетности по успешным платежам Партнер запрашивает направляет менеджеру e-mail адрес для получения реестров.

## Формат отчета

Отчет прикрепляется в виде вложения к письму, направляемому на адрес, указанный Партнером.

**Общее описание**

Маска наименования файла:

* + - ежедневный отчет – XXXXX\_DDMMYYYY.csv, где XXXXX – краткое наименование партнера в Сервисе, DDMMYYYY – день за который сформирован отчет;
    - ежемесячный отчет - XXXXX\_MMYYYY.csv, где XXXXX – краткое наименование партнера в Сервисе, MMYYYY – месяц за который сформирован отчет.

Формат CSV, разделитель «;», кодировка UTF8

Мультиплатежи в отчете отображаются потранзакционно, каждое начисление отдельной строкой.

**Заголовок отчета**

Первая строка отчета - заголовок, содержит информацию о самом реестре и состоит из следующих полей:

|  |  |  |
| --- | --- | --- |
| **Название** | **Описание** | **Значение** |
| strNo | Номер строки | int |
| reestrDate | Дата и время формирования отчета | DD.MM.YYYY HH:MM:SS |
| reportDate | Период, за который сформирован отчет | DD.MM.YYYY или MM.YYYY |
| quantityTotal | Количество платежей в отчете | int |
| amountTotal | Сумма платежей в отчете, в копейках | int |

**Тело отчета**

|  |  |  |
| --- | --- | --- |
| **Название** | **Описание** | **Значение** |
| strNo | Номер строки | int |
| orderID | Уникальный номер заказа на стороне Партнера | string |
| depType | Наименование ведомства Получателя | string |
| payType | Способ оплаты | string |
| paymentID | Идентификатор Платежа в ПЦ | int |
| paymentDate | Дата и время оплаты | DD.MM.YYYY HH:MM:SS |
| supplierBillID | Уникальный идентификатор Начисления (ГИС ГМП) | string |
| paymentDocumentID | Идентификатор платежного документа (ГИС ЖКХ) | string |
| amount | Сумма Платежа в пользу Получателя, в копейках. | int |
| uip | Уникальный Идентификатор Платежа. Для авторизованных платежей, импортированных в ГИС ГМП | string |
